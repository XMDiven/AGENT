  • Agent 已是多轮循环，不再是「单步编排」：run_agent_loop() 已落地——模型每轮自主决定调工具还是直接作答，工具结果以 ToolMessage 回灌供下一轮决策，最多 4 轮、超预算强制收敛（termination_reason 区分 final_answer / max_steps / failed）。下文凡「单步编排、不是多轮 ReAct」的话术一律改讲「ReAct 式多轮循环的 function calling 实现」；新的诚实边界是无跨会话长期 memory、每轮只执行一个工具调用。
  • hybrid search 已做完，结论是第二个「负结果」：BM25 + 向量 RRF 融合已实现（opt-in），实测全集 27 条上 dense 仍占优（MRR 0.901 vs 最佳 hybrid 0.873），仅 8 条代码符号难例子集上 hybrid 以 0.3/0.7 权重略胜（0.562 vs 0.525）；过程中发现并修复 BM25 默认分词对代码符号零召回的问题（code-aware 分词器，修复后难例进 Top-2）。下文凡「hybrid 没做、是后续方向」的话术，一律替换成这个实验故事。
  • web_search 工具已决策「不做」：代码里没有、简历里也已删。下文任何 web_search / 网络搜索话术整段忽略、绝不主动提；被问到就答「评估后认为对受控知识库问答价值不足、且引入注入与治理成本，故没有引入外部搜索依赖」。
  • 检索优化的真实结论是「负结果」，不是「rerank 提升」：在 27 条 golden set（含 3 条难例）上实测 MMR，每个 λ 下 MRR 都回退（similarity 0.901 vs MMR 0.892@λ0.3），唯一收益是多样性且 judge 无提升，所以默认保留 similarity、MMR 仅作可配置 opt-in。下文凡「做了 rerank 把命中率从 A 提到 B」的话术，一律替换成这个「用数据否决了自己的改动」的故事——它比假装提升更有说服力。
三张王牌（面试主动打出去）：① model_validator 强制评分总评由四维分数推导、机制上防止 judge 自评注水；② MMR、hybrid 两个负结果实验 → 数据驱动保留 similarity；③ 多轮 agent loop 的受控设计（步数预算、强制收敛、失败结构化回灌、token 裁剪）。

你这个项目是做什么的？ 从 RAG 知识库问答演进到轻量 Agent；覆盖 ingestion、retrieval、generation、evaluation；不是只调 API。 为什么不是普通聊天机器人？为什么需要知识库？
完整讲一下 /ask 链路。 router -> AskRequest -> ask_question() -> analysis -> plan -> Qdrant retriever -> context -> LLM -> answer/sources/trace。 documents 为空怎么办？documents 不相关怎么办？

为什么要有 sources？ 答案溯源、降低黑盒感、方便判断错误来自检索还是生成；边界是 sources 不代表一定完整或最优。 如何保证答案里的引用真实对应 sources？
为什么要有 trace？ 记录问题分析、检索规划、检索数量、生成状态和耗时；用于调试、评估和定位问题。 它和正式日志、metrics、distributed tracing 的区别是什么？

你项目最大的不足是什么？ 没有 rerank/score threshold；评估集小；Agent 无跨会话 memory；没有生产级权限、监控和异步任务。hybrid 已实测——负结果，保留 similarity。 两周内你优先补什么？为什么？
哪些内容可以写进简历？哪些不能吹？ 能写 RAG、Qdrant、FastAPI、sources/trace、LLM-as-Judge、function calling 多工具 Agent、多轮 agent loop、hybrid 检索实验 ✅；不能吹生产级、长期记忆、多 Agent 协作、大规模评测。 你怎么证明这些不是只停留在 README？
25%：RAG / Vector DB / LLM
可能问题 标准回答必须覆盖 可能继续追问
什么是 RAG？ 先检索外部知识，再让 LLM 基于 context 生成；优势是私有知识、更新知识、降低幻觉；不能完全消除幻觉。 RAG 和 fine-tuning 的区别？
chunk size 和 overlap 怎么选？ chunk size 是语义完整性和噪声/token 成本的折中；overlap 防止边界断裂；要通过 eval 调参。 为什么你现在是 800/100？依据够吗？
embedding 是什么？ 文本到向量的语义表示；相似语义在向量空间更近；不是完美理解，专有名词和数字可能需要 hybrid。 换 embedding 模型后旧索引怎么办？
Qdrant / 向量数据库解决什么？ 存储向量和 payload；支持相似度搜索和 metadata filter；比普通数据库更适合高维向量检索。 collection、point、payload、vector 是什么？
top_k 为什么不是越大越好？ 太小漏证据，太大引入噪声、增加 token 和延迟；应该基于 Recall@K、groundedness、latency 评估。 为什么你的默认是 7？
检索为空怎么办？检索不相关怎么办？ 空结果直接 fallback；不相关更危险，需要 score threshold、rerank、query rewrite、评估失败样本。 你现在有没有 score threshold？
rerank 解决什么问题？ 向量检索负责初筛召回，rerank 负责更精细排序，提高最终上下文精度；代价是延迟和成本。 rerank 是提高 recall 还是 precision？
hybrid search 有什么价值？ 结合关键词和向量检索，改善专有名词、代码符号、编号、API 名称的召回。 如果文档里有大量 API 名称，你会怎么改？
RAG 如何减少幻觉？ 用 context 约束回答，用 Prompt 要求只基于证据，用 sources 溯源，用 judge/评估检查 groundedness。 如果模型还是编造怎么办？
Prompt A/B 怎么评估？ 固定问题集、检索配置、模型和 judge，只改变 Prompt；看质量、格式、延迟、成本；样本小不能外推。 v2 质量更好但慢，你怎么选？
15%：Agent / Tool Calling
可能问题 标准回答必须覆盖 可能继续追问
Agent 和 RAG 的区别是什么？ RAG 解决知识问答；Agent 解决工具选择、执行动作、状态管理和多步任务；你的 RAG 是 Agent 的工具。 什么场景只用 RAG 就够？
你的 Agent 链路怎么走？ /agent/run -> run_agent() -> analyze_query() -> plan_tool() -> execute_plan() -> ToolResult/trace。 Agent trace 和 RAG trace 区别？
planner 为什么是规则式？ 工具少、问题类型简单，规则式更可控、可测、成本低；LLM planner 后续可引入，但会增加不确定性。 规则式 planner 泛化弱怎么办？
什么是 tool calling？ 让模型或编排层调用外部函数/API；关键是工具 schema、参数校验、返回结构和权限控制。 为什么不能让模型随便调用工具？
executor 负责什么？ 根据计划执行工具，包装 ToolResult，记录 status、output、attempts，并处理失败。 为什么 retrieval tool 可以 retry？
retry 有什么风险？ 读操作 retry 风险较低；写操作可能产生重复副作用，要有幂等 key、错误分类、最大次数和退避。 哪些错误不应该 retry？
你的 Agent 是 ReAct 吗？ 是 ReAct 式的多轮循环：模型每轮决定调工具还是作答，工具结果回灌后再决策，最多 4 轮、超预算强制收敛。区别是用 native function calling 实现、没有显式 thought 文本。 怎么升级成 LangGraph？和文本式 ReAct 比有什么取舍？
summary_tool 是真摘要吗？ 是真 LLM 摘要（空输入/空输出兜底 + 长度上限）；早期用字符串截断占位先验证编排路径，后替换成 LLM 实现——这条演进路径本身就值得讲。 摘要质量怎么评估？
Agent 需要 memory 吗？ agent loop 在单次 run 内用消息历史维护多轮工具调用上下文；跨会话长期 memory 没做，它会带来隐私、压缩、状态清理和一致性问题。 长期 memory 怎么评估是否污染？
如何评估 Agent 是否做对？ 看工具选择是否合理、工具输入是否正确、执行是否成功、最终答案是否完成目标；需要 trace 和任务级 golden cases。 只评估最终答案够不够？
15%：Python / FastAPI / 后端
可能问题 标准回答必须覆盖 可能继续追问
router、service、schema、infrastructure 为什么分层？ router 处理 HTTP；schema 校验数据；service 编排业务；infrastructure 封装外部依赖。价值是职责清楚、方便测试和替换。 如果都写 router 里有什么问题？
Pydantic 有什么用？ 定义请求/响应结构、类型校验、序列化、自动文档；但业务规则仍要自己设计。 dataclass 和 Pydantic 区别？
/ask 和 /ask/stream 区别？ 普通接口完整生成后返回；stream 用 NDJSON 分块返回 answer_delta、sources、trace、done，降低等待感。 流式是不是能减少总耗时？
FastAPI async 一定更快吗？ 不一定。async 适合 I/O 等待；如果内部是阻塞库或 CPU 密集任务，仍会阻塞，需要异步客户端、线程池或任务队列。 PDF 解析应该 async 吗？
接口失败怎么返回？ 区分参数错误、资源不存在、外部服务失败和内部异常；返回明确状态码和错误信息；内部堆栈不要暴露给用户。 LLM 超时用 500 还是 503？
上传接口有什么安全点？ 限制扩展名、文件大小、MIME；用安全文件名防路径穿越；避免覆盖；真实场景要扫描和异步处理。 Path(filename).name 解决什么？
如何测试依赖 LLM/Qdrant 的代码？ 用 mock/fake/monkeypatch 替换外部调用；测试 service 流程和边界，不依赖网络和 API key。 单元测试和集成测试怎么分？
Python generator 有什么用？ 按需产生数据，适合流式响应、大文件处理、长序列迭代；避免一次性加载全部内容。 yield 和 return 区别？
异常处理中为什么不要裸 except？ 会吞掉真实错误，难排查；应该捕获具体异常，记录 error type 和上下文，必要时降级或重新抛出。 什么时候可以捕获 Exception？
你会如何 debug 一个 API 慢的问题？ 用 trace 拆分 router/service/retrieval/generation 耗时；看外部服务、token 数、重试、网络和日志。 p95 latency 比平均值重要在哪里？
15%：评估 / 可观测 / 生产化
可能问题 标准回答必须覆盖 可能继续追问
你怎么证明回答质量变好了？ 固定 golden set；分开评估 retrieval 和 generation；记录通过率、groundedness、失败样本、延迟。 为什么不能只看平均分？
LLM-as-Judge 有什么价值和风险？ 价值是低成本批量评估；风险是偏置、不稳定、受格式影响、可能用外部知识；要 schema 校验和人工抽检。 Judge sources 不独立怎么办？
golden set 怎么设计？ 覆盖常见、边界、无答案、比较、长文档、易混淆问题；最好有 expected source 或答案要点。 27 个 case 够吗（含 3 条实测 hard case）？
trace 有什么生产化价值？ 定位每一步：query analysis、retrieval、generation、tool execution；能看耗时、状态、错误和输入输出摘要。 trace 和日志、metrics 有什么区别？
上线后要监控哪些指标？ latency、error rate、QPS、token、retrieval count、empty rate、fallback rate、judge score、tool failure rate。 哪些指标最能说明质量问题？
Prompt 更新怎么回滚？ Prompt 版本化；上线前跑 golden set；保留旧版本；线上观察失败率和延迟；异常时切回旧版本。 Prompt 也需要 CI 吗？
如何处理 LLM API 限流和超时？ 设置 timeout、retry/backoff、降级 fallback、错误分类、监控告警；高并发时加队列或限流。 哪些错误不该 retry？
为什么说你的项目还不是生产级？ 缺权限、异步队列、监控告警、安全防护、大规模评估、SLO、灰度发布；但已有可验证工程基线。 如果只补一个生产化模块，你补什么？
如何做 latency benchmark？ 固定问题集和环境，记录 retrieval/generation/total time，统计平均、p95、max；同时记录 token 和失败情况。 为什么外部 LLM benchmark 不稳定？
如何收集用户反馈？ 点赞点踩、追问、人工标注、失败反馈、fallback 触发；注意隐私脱敏和样本闭环。 用户点赞能直接当质量标签吗？

最终版能力的实现深问（写了就会被追问）

面试官看到 function calling、rerank、async 修复这些能力，一定会追问「怎么实现的」。下面是实现级问答；🔨 表示对应能力是路线图项，做完再这样答，✅ 表示已落地。
function calling 端到端怎么落地的？✅
  1. Q：tool schema 怎么来、怎么传给模型？

    答：复用 registry 里每个工具的 input_schema（JSON Schema）作为 tool/function 的 parameters，连同 name、description 一起放进 chat completions 的 tools 参数。我没有为 function calling 单独维护一套 schema，契约只有一份。

  2. Q：模型返回 tool_call 后你怎么处理？

    答：解析 tool_calls 里的工具名和 JSON 参数 → 用工具名从 registry 取工具 → 校验参数 → 执行 → 把结果和「模型选了哪个工具、填了什么参数」写回 trace。工具名不存在或参数非法时不执行，走 fallback。

  3. Q：模型不返回 tool_call / 选了不存在的工具怎么办？

    答：这正是我保留规则式 planner 作 fallback 的原因。模型超时、不支持 tool calling、或返回非法工具名时降级到规则式选工具，保证链路不断——可控性优先。

rerank：cross-encoder 还是 LLM 重排？怎么选？🔨
  1. Q：两者区别？

    答：cross-encoder（如 bge-reranker）把 query 和候选片段一起编码打分，精度高、延迟成本可控、可离线部署；LLM 重排灵活但慢且贵、不稳定。实习项目我倾向可插拔的 cross-encoder，或先用 retriever 的 similarity score 做阈值过滤，先拿到 before/after 数据。

  2. Q：rerank 提升 recall 还是 precision？

    答：主要提升 precision（让最相关片段排前、过滤噪声），不增加召回。所以是「高召回初筛 + 精排」两段式：初筛保 recall、rerank 保 precision。

  3. Q：怎么证明 rerank 有用？

    答：固定 golden set，对比开/关的 source hit rate / MRR / groundedness / latency，把 before/after 写进报告并说明样本量——不写没有数据支撑的百分比。我已经用这套方法做过一次：在 27 条 golden set 上对比 similarity 与 MMR，结果 MMR 每个 λ 下 MRR 都回退，所以我没把它设为默认、保留了 similarity。这就是「有数据就敢否决自己的改动」。

结构化输出：function calling / JSON mode / 手动解析怎么选？
  1. 答：需要模型「选操作 + 填参数」用 function calling；只需要一个固定结构 JSON 用 JSON mode + Pydantic 校验（我的 LLM-as-Judge 就是 Pydantic schema 校验 + 容错解析）；都不支持才退到手动解析 + 严格校验。核心是永远在边界做 schema 校验，不信模型直接给的结构。

  2. Q：模型返回的 JSON 不合法怎么办？

    答：我的 judge 链路就处理了：先 strip 掉 markdown code fence，再 model_validate_json，并用 model_validator 强制 overall_pass 由分数推导。生产里还会加重试或 repair 提示。

async def 路由里跑阻塞 I/O 会怎样？你怎么修的？✅
  1. Q：后果是什么？

    答：async def 路由跑在事件循环线程上，里面如果同步阻塞（LLM/Qdrant 网络调用）会卡住整个事件循环，并发下其他请求一起被拖死。这是 FastAPI 最常见的性能坑。

  2. Q：sync def 还是 run_in_threadpool？

    答：看路由体内有没有真正的 await。我的路由全是同步调用、没有 await，所以最诚实的修法是改成 def，FastAPI 自动丢线程池——而不是假装 async。只有真有异步调用时才用 await run_in_threadpool(...) 包阻塞部分。

  3. Q：流式接口呢？

    答：StreamingResponse 包同步 generator 时 Starlette 会用线程池迭代，所以流式没有 /ask、/ingest 那么直接的阻塞；我还把检索 retry 统一到流式路径，保证两个接口健壮性一致。

为什么没做 web_search / 联网工具？✅（边界判断）
  1. 答：我评估后主动决定不引入外部搜索：本项目定位是「基于受控知识库的 grounded 问答」，引入开放网络会带来注入风险、结果不可控，还要额外做超时/失败/配额治理，对一个要讲清边界的实习项目是负资产。把检索、摘要、拆解这条受控工具链做扎实，比堆一个不可控的联网工具更有说服力——这本身就是工程取舍判断。

1. 两数之和

思路：哈希表存「值→下标」，遍历时查 target-x 是否已出现。

def twoSum(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        if target - x in seen:
            return [seen[target - x], i]
        seen[x] = i

复杂度：时间 O(n)，空间 O(n)

11. 盛最多水的容器

思路：左右双指针，面积由较矮一侧决定，移动较矮的指针。

def maxArea(height):
    l, r, ans = 0, len(height) - 1, 0
    while l < r:
        ans = max(ans, min(height[l], height[r]) * (r - l))
        if height[l] < height[r]:
            l += 1
        else:
            r -= 1
    return ans

复杂度：时间 O(n)，空间 O(1)

15. 三数之和

思路：排序后固定 i，剩下用左右双指针找和为 -nums[i]，注意跳重复。

def threeSum(nums):
    nums.sort()
    n, res = len(nums), []
    for i in range(n - 2):
        if nums[i] > 0:
            break
        if i > 0 and nums[i] == nums[i - 1]:
            continue
        l, r = i + 1, n - 1
        while l < r:
            s = nums[i] + nums[l] + nums[r]
            if s < 0:
                l += 1
            elif s > 0:
                r -= 1
            else:
                res.append([nums[i], nums[l], nums[r]])
                l += 1
                r -= 1
                while l < r and nums[l] == nums[l - 1]:
                    l += 1
                while l < r and nums[r] == nums[r + 1]:
                    r -= 1
    return res

复杂度：时间 O(n^2)，空间 O(1)（不计排序/输出）

26. 删除有序数组中的重复项

思路：慢指针 k 指向下一个待写入位置，快指针扫描去重。

def removeDuplicates(nums):
    if not nums:
        return 0
    k = 1
    for i in range(1, len(nums)):
        if nums[i] != nums[k - 1]:
            nums[k] = nums[i]
            k += 1
    return k

复杂度：时间 O(n)，空间 O(1)

def moveZeroes(nums):

            nums[i], nums[j] = nums[j], nums[i]

167. 两数之和 II（输入有序）

思路：数组有序，直接左右双指针向中间逼近。

def twoSum(numbers, target):
    l, r = 0, len(numbers) - 1
    while l < r:
        s = numbers[l] + numbers[r]
        if s == target:
            return [l + 1, r + 1]
        if s < target:
            l += 1
        else:
            r -= 1

复杂度：时间 O(n)，空间 O(1)

88. 合并两个有序数组

思路：从后往前填，避免覆盖 nums1 的有效元素。

def merge(nums1, m, nums2, n):
    i, j, k = m - 1, n - 1, m + n - 1
    while j >= 0:
        if i >= 0 and nums1[i] > nums2[j]:
            nums1[k] = nums1[i]
            i -= 1
        else:
            nums1[k] = nums2[j]
            j -= 1
        k -= 1

复杂度：时间 O(m+n)，空间 O(1)

3. 无重复字符的最长子串

思路：滑动窗口 + 哈希记录每个字符最近下标，重复时左边界跳到其后。

def lengthOfLongestSubstring(s):
    last = {}
    left = ans = 0
    for i, c in enumerate(s):
        if c in last and last[c] >= left:
            left = last[c] + 1
        last[c] = i
        ans = max(ans, i - left + 1)
    return ans

复杂度：时间 O(n)，空间 O(min(n, 字符集))

76. 最小覆盖子串

思路：滑动窗口 + 计数：missing 记录还差几个字符，凑齐后收缩左边界更新最优。

from collections import Counter

def minWindow(s, t):
    need = Counter(t)
    missing = len(t)
    l = 0
    best = (float('inf'), 0, 0)
    for r, c in enumerate(s):
        if need[c] > 0:
            missing -= 1
        need[c] -= 1
        while missing == 0:
            if r - l + 1 < best[0]:
                best = (r - l + 1, l, r + 1)
            need[s[l]] += 1
            if need[s[l]] > 0:
                missing += 1
            l += 1
    return s[best[1]:best[2]] if best[0] < float('inf') else

复杂度：时间 O(|s|+|t|)，空间 O(字符集)

239. 滑动窗口最大值

思路：单调递减双端队列存下标，队首即窗口最大值。

from collections import deque

def maxSlidingWindow(nums, k):
    dq, res = deque(), []
    for i, x in enumerate(nums):
        while dq and nums[dq[-1]] <= x:
            dq.pop()
        dq.append(i)
        if dq[0] <= i - k:
            dq.popleft()
        if i >= k - 1:
            res.append(nums[dq[0]])
    return res

复杂度：时间 O(n)，空间 O(k)

53. 最大子数组和

思路：Kadane：cur = max(x, cur + x)，全程记录最大值。

def maxSubArray(nums):
    cur = ans = nums[0]
    for x in nums[1:]:
        cur = max(x, cur + x)
        ans = max(ans, cur)
    return ans

复杂度：时间 O(n)，空间 O(1)

56. 合并区间

思路：按起点排序，与结果最后一个区间重叠则合并，否则新开。

def merge(intervals):
    intervals.sort()
    res = []
    for s, e in intervals:
        if res and s <= res[-1][1]:
            res[-1][1] = max(res[-1][1], e)
        else:
            res.append([s, e])
    return res

复杂度：时间 O(n log n)，空间 O(n)

238. 除自身以外数组的乘积

思路：先存前缀积，再乘后缀积，O(1) 额外空间（输出不计）。

def productExceptSelf(nums):
    n = len(nums)
    res = [1] * n
    pre = 1
    for i in range(n):
        res[i] = pre
        pre *= nums[i]
    suf = 1
    for i in range(n - 1, -1, -1):
        res[i] *= suf
        suf *= nums[i]
    return res

复杂度：时间 O(n)，空间 O(1)

41. 缺失的第一个正数

思路：原地哈希：把值 x 放到下标 x-1，再扫第一个不匹配的位置。

def firstMissingPositive(nums):
    n = len(nums)
    for i in range(n):
        while 1 <= nums[i] <= n and nums[nums[i] - 1] != nums[i]:
            j = nums[i] - 1
            nums[i], nums[j] = nums[j], nums[i]
    for i in range(n):
        if nums[i] != i + 1:
            return i + 1
    return n + 1

复杂度：时间 O(n)，空间 O(1)

125. 验证回文串

思路：双指针，跳过非字母数字，比较小写。

def isPalindrome(s):
    l, r = 0, len(s) - 1
    while l < r:
        while l < r and not s[l].isalnum():
            l += 1
        while l < r and not s[r].isalnum():
            r -= 1
        if s[l].lower() != s[r].lower():
            return False
        l += 1
        r -= 1
    return True

复杂度：时间 O(n)，空间 O(1)

5. 最长回文子串

思路：中心扩展：以每个字符/间隙为中心向两侧扩。

def longestPalindrome(s):
    def expand(l, r):
        while l >= 0 and r < len(s) and s[l] == s[r]:
            l -= 1
            r += 1
        return l + 1, r - 1
    start, end = 0, 0
    for i in range(len(s)):
        for l, r in (expand(i, i), expand(i, i + 1)):
            if r - l > end - start:
                start, end = l, r
    return s[start:end + 1]

复杂度：时间 O(n^2)，空间 O(1)

49. 字母异位词分组

思路：以「排序后的字符串」作为哈希 key 分组。

from collections import defaultdict

def groupAnagrams(strs):
    groups = defaultdict(list)
    for s in strs:
        groups[tuple(sorted(s))].append(s)
    return list(groups.values())

复杂度：时间 O(n·k log k)，空间 O(n·k)

680. 验证回文串 II

思路：双指针，遇到不等时尝试删左或删右各一次。

def validPalindrome(s):
    def ok(l, r):
        while l < r:
            if s[l] != s[r]:
                return False
            l += 1
            r -= 1
        return True
    l, r = 0, len(s) - 1
    while l < r:
        if s[l] != s[r]:
            return ok(l + 1, r) or ok(l, r - 1)
        l += 1
        r -= 1
    return True

复杂度：时间 O(n)，空间 O(1)

6. Z 字形变换

思路：按行模拟，方向在顶/底反转。

def convert(s, numRows):
    if numRows == 1:
        return s
    rows = [''] * numRows
    cur, step = 0, 1
    for c in s:
        rows[cur] += c
        if cur == 0:
            step = 1
        elif cur == numRows - 1:
            step = -1
        cur += step
    return ''.join(rows)

复杂度：时间 O(n)，空间 O(n)

def isBipartite(graph):
    color = {}
    for s in range(len(graph)):
        if s in color:
            continue
        color[s] = 0
        q = deque([s])
        while q:
            u = q.popleft()
            for v in graph[u]:
                if v not in color:
                    color[v] = color[u] ^ 1
                    q.append(v)
                elif color[v] == color[u]:

class Trie:
    def __init__(self):
        self.children = {}
        self.end = False

    def insert(self, word):
        node = self
        for c in word:
            node = node.children.setdefault(c, Trie())
        node.end = True

    def _find(self, prefix):
        node = self
        for c in prefix:
            if c not in node.children:
                return None
            node = node.children[c]
        return node

    def search(self, word):
        node = self._find(word)
        return node is not None and node.end

    def startsWith(self, prefix):

            x = parent[x]

def solveNQueens(n):
    res = []
    cols, d1, d2 = set(), set(), set()
    board = [['.'] * n for _ in range(n)]
    def bt(r):
        if r == n:
            res.append([''.join(row) for row in board])
            return
        for c in range(n):
            if c in cols or (r - c) in d1 or (r + c) in d2:
                continue

            cols.discard(c); d1.discard(r - c); d2.discard(r + c)
    bt(0)
    return res

项目和简历防守：标准答案
  1. Q：你的项目解决什么问题？为什么从 RAG 演进到 Agent？

    答：这个项目先解决知识库问答问题，也就是让用户基于本地 Markdown/PDF 文档得到带来源的回答。RAG 负责检索和生成，Agent 是在 RAG 之上加一层轻量编排，用来做问题分析、工具选择、执行状态和 trace。这样项目不是只停留在“问答 demo”，而是开始向可扩展的 AI 应用后端演进。

  2. Q：完整讲一下 /ask 请求链路。

    答：/ask 请求进入 FastAPI router 后解析成 AskRequest，然后调用 service 层的 ask_question()。service 先用 analyze_query() 分析问题类型，再用 plan_retrieval() 生成检索计划。如果需要检索，就通过 Qdrant retriever 拿到 documents，再用 format_context() 拼成上下文，交给 Prompt 和 LLM 生成答案，最后返回 answer、sources 和 trace。

  3. Q：文档从上传到入库怎么走？Markdown 和 PDF 有什么区别？

    答：上传到入库我拆成了两步：上传接口先做文件类型校验和安全文件名处理，然后把 Markdown 或 PDF 保存到 data/raw；真正入库时，再由 /documents/ingest 根据文件后缀选择对应的解析和切分流程，生成 chunks 后通过 embedding 写入 Qdrant。Markdown 和 PDF 的区别主要在元数据来源：Markdown 有标题结构，所以可以保留章节层级；PDF 更偏页面级解析，通常记录页码和来源文件。最后两者都会统一成标准 chunks，所以后面的检索、生成和 sources 返回可以复用同一套 RAG 链路。

  4. Q：为什么要返回 sources 和 trace？

    答：sources 用来做答案溯源，让用户知道回答依据；trace 用来记录问题分析、检索规划、检索和生成过程，方便调试和评估。它们的工程价值是降低黑盒感，并且能定位答案错误到底来自检索、Prompt 还是 LLM。边界是当前 trace 还不是完整生产级观测系统。

RAG、向量检索和数据入库：标准答案
  1. Q：什么是 RAG？它比直接问 LLM 好在哪里？

    答：RAG 是 Retrieval-Augmented Generation，先从外部知识库检索相关上下文，再交给 LLM 生成回答。它的优势是能接入私有知识和最新知识，并降低模型凭空编造的概率。边界是如果检索错了，LLM 仍然可能基于错误上下文生成错误答案。

  2. Q：chunk size 和 overlap 怎么选？

    答：chunk size 是召回粒度和上下文完整性的折中。太小容易丢语义，太大容易引入噪声并增加 token 成本；overlap 用来减少切分边界丢信息，但太大会造成重复内容和索引膨胀。最好通过 golden set 对不同参数做 retrieval eval。

  3. Q：Markdown、PDF、网页、表格的切分策略有什么不同？

    答：Markdown 可以利用标题层级切分；PDF 通常先按页或段落抽取，再补 page metadata；网页要处理导航、广告和正文提取；表格要保留行列语义。核心原则是让 chunk 既能独立表达含义，又能保留足够 metadata 方便溯源。

  4. Q：embedding 是什么？为什么语义相似可以用向量距离表示？

    答：embedding 是把文本映射成高维向量，让语义相近的文本在向量空间里更接近。检索时用 cosine、dot product 等相似度找最接近的问题或文档片段。它不是完美理解语言，专有名词、数字和领域术语仍可能需要关键词检索或 rerank 补充。

  5. Q：向量数据库解决什么问题？为什么不用普通数据库全文扫描？

    答：向量数据库负责存储向量、metadata 和高效近似最近邻检索，适合语义相似度查询。普通数据库更擅长结构化查询和事务，不适合对大量高维向量做高效相似度搜索。真实系统通常会把向量检索和 metadata filter 结合起来。

  6. Q：collection、point、payload、vector 分别是什么？

    答：collection 可以理解为一组向量数据的集合；point 是一条向量记录；vector 是 embedding 后的数值向量；payload 是和向量绑定的 metadata，例如 source、page、section。面试时可以类比：vector 用于相似度检索，payload 用于过滤和溯源。

  7. Q：upsert 和 insert 有什么区别？为什么入库要考虑幂等？

    答：insert 通常表示新增，upsert 表示存在则更新、不存在则新增。RAG 入库要考虑幂等，是因为同一个文档可能重复构建索引；如果没有稳定 chunk ID，就会产生重复向量，影响检索质量和存储成本。

  8. Q：top_k 为什么不是越大越好？

    答：top_k 太小可能漏掉关键证据，太大则会引入无关 chunk，增加 token 成本和延迟，并可能污染 LLM 上下文。它应该通过 retrieval eval 选择，而不是拍脑袋固定。你项目里可以说当前 top_k=7 是小规模 golden set 下的经验折中。

  9. Q：如果检索为空怎么办？如果检索不相关怎么办？

    答：检索为空时应该返回 fallback，不要让 LLM 无依据硬答。更难的是检索不为空但不相关，这时需要相关性阈值、score 过滤、rerank 或更好的 query rewrite。面试中要强调“无检索”和“低质量检索”是两个问题。

  10. Q：similarity search、hybrid search、rerank 分别解决什么问题？

    答：similarity search 用向量相似度召回语义相关片段；hybrid search 结合关键词和向量，适合专有名词、编号、代码 API；rerank 在召回后重新排序，提高最终上下文精度。常见做法是先高召回，再 rerank 提精度。我自己实测过 hybrid：全集上 dense 仍占优，只在代码符号难例上 hybrid 略胜——它不是无脑赢，要看语料和查询分布。

  11. Q：MMR 是什么？它和普通 top_k 有什么不同？

    答：MMR 会同时考虑相关性和多样性，避免返回的结果都高度重复。普通 top_k 只按相似度排序，可能多个 chunk 都来自同一段内容。比较类、多来源类问题通常更需要多样性。

  12. Q：parent-child chunk / small-to-big retrieval 有什么价值？

    答：小 chunk 适合精确召回，大 chunk 或 parent document 适合提供完整上下文。small-to-big retrieval 的思路是先用小片段匹配，再回填更大的上下文，平衡检索精度和回答完整性。

  13. Q：query rewrite、multi-query retrieval、HyDE 分别解决什么问题？

    答：query rewrite 用来把用户问题改写成更适合检索的形式；multi-query 用多个不同角度的问题提高召回；HyDE 先生成一个假设答案再检索相似文档。它们都在解决“用户原始问题不一定适合直接检索”的问题。

  14. Q：如何评估 retrieval quality？Recall@K、MRR、context precision 怎么理解？

    答：Recall@K 看期望证据是否出现在前 K 个结果里；MRR 看第一个正确结果排得有多靠前；context precision 看返回上下文里有多少是真相关。RAG 评估要把检索质量和生成质量分开，否则不知道问题出在哪一层。

  15. Q：如果用户上传 1000 个 PDF，入库流程怎么设计？

    答：不应该让 HTTP 请求同步完成全部解析和 embedding。更合理是上传后生成 job id，后台任务异步解析、切分、批量 embedding、写入向量库，并提供任务状态查询。还要考虑失败重试、文件大小限制、重复入库和增量更新。

  16. Q：如果更新文档，旧向量怎么处理？

    答：常见做法是根据 document_id 或稳定 chunk_id 删除旧 chunk 后重建，或者做版本化索引。关键是不要让旧版本和新版本同时被检索到，除非业务明确需要保留历史版本。

  17. Q：多租户知识库如何隔离检索范围？

    答：至少要在 metadata 中记录 tenant_id、user_id、workspace_id，并在检索时强制 filter。更严格的做法是按租户拆 collection 或做权限服务。不能只靠 Prompt 告诉模型不要访问别人的文档。

LLM、Prompt 和结构化输出：标准答案
  1. Q：token、context window、max_tokens 是什么？

    答：token 是模型处理文本的基本单位；context window 是模型一次能看到的最大上下文；max_tokens 限制输出长度。它们直接影响成本、延迟和是否会截断上下文。RAG 中要控制检索片段数量和 Prompt 长度。

  2. Q：temperature、top_p 影响什么？

    答：temperature 和 top_p 都影响输出随机性。问答、抽取、评估这类需要稳定性的任务通常调低；创意写作可以适当调高。面试中要强调参数不是越高越聪明，而是取决于任务对稳定性和多样性的要求。

  3. Q：system/user/assistant message 有什么区别？

    答：system message 放全局角色和强约束；user message 是用户输入；assistant message 是模型历史回答。RAG 中通常把“只基于上下文回答、引用格式、不知道就说不知道”这类规则放在 system 或固定 Prompt 中，而不是依赖用户自觉。

  4. Q：Prompt 在 RAG 链路里起什么作用？

    答：Prompt 把用户问题、检索上下文和回答约束组织成模型可执行的指令。它决定模型如何使用 context、如何引用来源、遇到证据不足时怎么回答。边界是 Prompt 不能弥补严重错误的检索结果。

  5. Q：为什么 Prompt 要要求“只基于 context 回答”？

    答：这是为了把回答约束在知识库证据内，降低模型使用外部知识或编造信息的风险。它对 grounded answer 很重要，但不能完全防止幻觉，所以还需要 sources、judge、人工抽检和相关性过滤。

  6. Q：RAG 能完全消除幻觉吗？

    答：不能。RAG 能降低幻觉风险，因为它提供外部证据，但如果检索结果不相关、上下文缺失、Prompt 约束弱或模型不遵循指令，仍然会产生错误答案。因此要配合检索评估、来源引用、低置信 fallback 和 LLM-as-Judge。

  7. Q：结构化输出为什么重要？如果 JSON 不稳定怎么办？

    答：结构化输出方便下游程序解析和自动评估。提高稳定性的方式包括明确 schema、给示例、使用模型原生 structured output、Pydantic 校验、解析失败重试或降级。不要只靠“请输出 JSON”这一句话。

  8. Q：Prompt A/B 测试怎么做才可信？

    答：要固定问题集、检索配置、模型和评估标准，只改变 Prompt；同时记录质量指标和延迟成本。样本太小时只能作为趋势参考，不能说全局最优。你项目可以说已做小规模 Prompt 版本对比，但还不是完整平台。

  9. Q：v2 Prompt 质量更高但延迟更高，怎么决策？

    答：要看业务目标。如果是通用问答，质量提升很小但延迟明显增加，可能继续用 v1；如果是需要固定格式或高完整性的场景，可以选择 v2。工程决策不是只看单一分数，而是质量、延迟、成本和用户场景的权衡。

  10. Q：什么是 prompt injection？怎么处理？

    答：prompt injection 是用户输入或检索文档试图覆盖系统指令，例如“忽略之前规则”。处理方式包括系统指令隔离、文档内容降权、工具权限控制、输出检测、敏感操作人工确认。不能只靠 Prompt 自己防御 Prompt injection。

Agent、Tool Calling 和工作流：标准答案
  1. Q：Agent 和 RAG 的区别是什么？

    答：RAG 主要解决“基于知识库回答问题”，核心是检索和生成；Agent 更强调根据目标选择工具、执行动作、维护状态和处理多步任务。简单知识问答通常 RAG 就够，涉及工具调用、任务分解和外部系统操作时才需要 Agent。

  2. Q：什么是 tool calling？为什么要定义工具 schema？

    答：tool calling 是让模型或编排层调用外部函数、API 或服务。工具 schema 明确工具名称、参数、类型和返回结构，能降低调用错误，并方便校验、审计和测试。没有 schema 的工具调用很难稳定扩展。

  3. Q：planner、executor、memory、tool registry、state 分别负责什么？

    答：planner 负责决定下一步做什么或用哪个工具；executor 负责执行工具并处理结果；memory 保存历史和长期信息；tool registry 管理可用工具；state 记录一次任务的中间状态。好的 Agent 设计要把决策、执行和状态分开。

  4. Q：你的 planner 为什么不用 LLM？

    答：我用 function calling 让模型按意图选工具 ✅，再用规则式 planner 作 fallback——双层兼顾泛化与可控。模型不支持 tool calling、超时或返回非法工具时由规则兜底，省 token 又可测。多轮决策由 agent loop 承接 ✅：模型基于上一轮工具结果决定下一步。再往上是 LangGraph 状态流。

  5. Q：ReAct 解决什么问题？有什么风险？

    答：ReAct 通过 thought/action/observation 循环，让模型边推理边调用工具，适合多步任务。风险是循环不可控、工具过度调用、推理过程泄露、错误观察导致后续动作偏离。因此需要最大步数、权限控制、trace 和失败策略。我的 agent loop 就是这类循环的 function calling 实现：最多 4 轮、失败结构化回灌、超预算强制收敛。

  6. Q：工具调用失败时应该 retry 吗？

    答：临时网络错误、限流、外部服务抖动适合 retry；参数错误、权限错误、业务校验失败通常不适合盲目 retry。retry 要配合最大次数、超时、退避和错误分类，否则会放大故障。

  7. Q：如何避免 Agent 无限循环？

    答：设置最大步数、最大工具调用次数、超时时间和终止条件；同时记录 trace，检查每一步是否有实际进展。对于高风险工具，还要加入人工审批或明确权限边界。

  8. Q：为什么不建议实习项目一上来做复杂多 Agent？

    答：多 Agent 会引入通信、状态、评估和调试复杂度。如果基础 RAG、工具契约和评估还不稳，复杂多 Agent 只是堆概念。更好的路线是先做好单 Agent 工具编排、trace 和 eval，再逐步扩展。

  9. Q：LangChain Agent 和 LangGraph 大致有什么差异？

    答：LangChain Agent 更偏快速搭建工具调用链路；LangGraph 更强调显式状态图、可控流程、checkpoint 和复杂工作流。面试时不用深讲源码，重点是说明复杂 Agent 需要状态管理和可观测性，而不是只靠模型自由发挥。

  10. Q：Agent trace 应该记录什么？

    答：至少记录用户目标、每一步选择的工具、输入摘要、输出状态、错误信息、attempts、耗时和最终结果。trace 的价值是复盘工具选择是否合理、失败发生在哪一步，以及为后续评估提供证据。

评估、A/B 和 LLM-as-Judge：标准答案
  1. Q：如何评估一个 RAG 系统？

    答：要分两层评估：检索层看 expected evidence 是否被召回，指标包括 Recall@K、MRR、context precision；生成层看 answer relevance、completeness、groundedness 和格式。两层分开评估，才能知道是检索错还是生成错。

  2. Q：faithfulness / groundedness 是什么？

    答：它们衡量答案是否能被给定上下文支持。一个答案可能语言流畅、相关性高，但如果关键结论不在 context 里，就不 grounded。RAG 项目里 groundedness 比普通聊天更重要，因为用户需要可信依据。

  3. Q：LLM-as-Judge 有什么价值和风险？

    答：价值是能低成本批量评估回答的相关性、完整性、事实支撑和格式，适合回归测试。风险是 judge 也可能偏置、不稳定、受格式影响或使用外部知识。因此要用固定 Prompt、结构化 schema、人工抽检和多轮对比。

  4. Q：如何设计 golden set？

    答：golden set 要覆盖常见问题、边界问题、比较类问题、无答案问题和容易混淆的问题。每个 case 最好有 expected source 或 expected answer points。样本不是越多越好，关键是能代表真实用户问题和高风险场景。

  5. Q：Prompt A/B 测试如何控制变量？

    答：固定问题集、语料、检索配置、模型和 judge 标准，只改变 Prompt。记录质量指标、延迟和成本，避免把外部服务波动误认为 Prompt 效果。样本小的时候只说“当前观察”，不要说“证明最优”。

  6. Q：如何做回归测试，避免新 Prompt 让老问题变差？

    答：保留一组稳定 golden cases，每次改 Prompt、检索参数或模型后自动跑评估，比较通过率、平均分、失败 case 和延迟。重点看失败样本，而不是只看平均分。

  7. Q：线上应该收集哪些反馈信号？

    答：可以收集点赞点踩、用户追问、fallback rate、empty retrieval rate、latency、token 成本、人工标注和投诉样本。要注意隐私和脱敏，不能把敏感用户内容无控制地记录进日志。

FastAPI、HTTP、Python 和测试：标准答案
  1. Q：router、service、schema、infrastructure 为什么要分层？

    答：router 处理 HTTP 请求响应，schema 做数据校验和响应结构，service 放业务流程，infrastructure 封装外部依赖如向量库和 LLM。分层的价值是职责清楚、测试容易、后续替换外部服务时影响小。

  2. Q：Pydantic 在请求和响应里起什么作用？

    答：Pydantic 用来定义数据结构、类型校验和序列化。FastAPI 可以根据 Pydantic model 自动解析 request body，并生成响应模型和文档。它能减少手写校验代码，但业务规则仍需要自己设计。

  3. Q：同步接口和异步接口有什么区别？

    答：异步适合 I/O 密集场景，可以在等待外部服务时释放事件循环处理其他请求；同步更简单。异步不等于一定更快，如果内部调用阻塞库或 CPU 密集任务，仍可能阻塞，需要线程池、任务队列或异步客户端配合。

  4. Q：常见 HTTP 状态码怎么用？

    答：200 表示成功，400 表示请求参数不合法，401 未认证，403 无权限，404 资源不存在，422 常见于请求体校验失败，500 是服务端异常。AI 应用还要区分外部 LLM 服务失败和本服务业务失败。

  5. Q：/ask 和 /ask/stream 有什么区别？

    答：/ask 等完整答案生成后一次性返回；/ask/stream 把生成过程中的 answer delta 分块返回，降低用户等待感。检索阶段通常仍要先完成，流式主要优化生成阶段体验。

  6. Q：上传接口如何避免路径穿越？

    答：不能直接信任用户传入的文件名，要用 Path(filename).name 去掉路径部分，并限制扩展名、文件大小和存储目录。更严格还要校验 MIME、扫描内容和避免覆盖已有文件。

  7. Q：如何 mock Qdrant 或 LLM 做单元测试？

    答：把外部依赖封装在 infrastructure 或工具函数里，测试时用 monkeypatch 或 fake client 替换真实调用。这样测试关注业务流程和边界，不依赖网络、API key 或外部服务状态。

  8. Q：generator 和 yield 适合什么场景？

    答：generator 可以按需产生数据，不需要一次性构建完整列表，适合流式响应、大文件处理和长序列迭代。你项目里的 stream answer 就可以用这种思想逐块返回内容。

  9. Q：dataclass 和 Pydantic model 有什么区别？

    答：dataclass 适合轻量内部数据结构，主要减少样板代码；Pydantic model 更适合 API 边界和外部输入输出，因为它有更强的校验、解析和序列化能力。内部状态可以用 dataclass，接口请求响应更适合 Pydantic。

  10. Q：异常处理应该怎么设计？

    答：能预期的业务错误要返回明确错误码和信息，外部服务错误要记录并降级，不能把内部堆栈直接暴露给用户。不要到处裸捕获 Exception 后吞掉错误，至少要记录 error type、trace 和上下文。

算法、CS 基础和 Coding：标准答案
  1. Q：现场 coding 你应该怎么开场？

    答：我会先复述题意和输入输出，确认边界 case；然后先给一个能工作的暴力思路，再优化到更合适的数据结构；写完后用 2-3 个例子手动走一遍，最后说明时间复杂度和空间复杂度。面试官看重的不只是答案，还看我是否能稳定拆题、写代码和验证。

  2. Q：两数之和怎么做？

    答：核心是用哈希表记录已经见过的数和下标。遍历数组时计算 complement = target - 当前值，如果 complement 已在表中，就返回两个下标；否则把当前值放入表。时间复杂度 O(n)，空间复杂度 O(n)。注意如果题目要求返回原始下标，就不要先排序破坏下标。

  3. Q：有效括号怎么做？

    答：用栈。遇到左括号入栈，遇到右括号就检查栈顶是否匹配；不匹配或栈为空就返回 false。遍历结束后栈为空才说明全部匹配。时间复杂度 O(n)，空间复杂度 O(n)。边界 case 是空字符串、只有右括号、嵌套括号和交叉括号。

  4. Q：二分查找最容易错在哪里？

    答：二分的关键是先定义搜索区间和循环不变量，比如闭区间 [left, right] 就用 left <= right，每次根据 mid 缩小区间。常见错误是边界更新不一致导致死循环或漏掉答案。时间复杂度 O(log n)，空间复杂度 O(1)。

  5. Q：滑动窗口适合什么题？

    答：滑动窗口适合连续子数组或子串问题，尤其是“最长/最短满足条件的连续区间”。套路是右指针扩张窗口、更新状态；当窗口不满足条件时移动左指针收缩。因为每个元素最多进出窗口一次，所以通常是 O(n)。

  6. Q：BFS 和 DFS 怎么区分？

    答：BFS 用队列按层遍历，适合无权图最短路径、层序遍历；DFS 用递归或栈深入搜索，适合连通性、回溯、路径枚举。图搜索都要注意 visited，避免重复访问或死循环。

  7. Q：简单动态规划怎么讲？

    答：DP 不要一上来背公式，而是按四步讲：状态定义、状态转移、初始条件、遍历顺序。比如爬楼梯可以定义 dp[i] 表示到第 i 阶的方法数，转移来自前一阶和前两阶。面试里简单 DP 重点是讲清“当前答案依赖哪些更小问题”。

  8. Q：时间复杂度和空间复杂度怎么分析？

    答：先定义输入规模 n，再看循环、递归和使用的数据结构如何随 n 增长。单层遍历通常 O(n)，嵌套遍历通常 O(n²)，二分通常 O(log n)，哈希表额外存 n 个元素就是 O(n) 空间。不要只说结论，要能指出复杂度来自哪一步。

  9. Q：GET 和 POST 有什么区别？

    答：GET 主要用于获取资源，参数通常在 URL 中，更容易被缓存和记录；POST 常用于提交数据或触发创建、计算等操作，请求体可以更复杂。严格说安全性不是由 GET/POST 本身决定的，仍然要靠 HTTPS、认证授权和服务端校验。

  10. Q：进程、线程、协程怎么回答最稳？

    答：进程是资源隔离单位，线程共享进程资源，协程是用户态调度的轻量并发。CPU 密集任务更依赖多进程或线程池，I/O 密集任务适合 async/await。FastAPI 的 async 主要价值是在等待 LLM、数据库、向量库这类 I/O 时提高并发能力。

  11. Q：数据库索引为什么能加速？

    答：索引通过额外数据结构减少全表扫描，让查询能更快定位候选数据。代价是写入变慢、占用更多存储，并且索引设计不合理也可能用不上。向量索引类似地是为了避免对所有向量暴力计算相似度，但它解决的是近似最近邻搜索，不等同于传统 B 树索引。

  12. Q：缓存能不能直接缓存 LLM 答案？

    答：可以缓存部分稳定问题，但不能无脑缓存。LLM 答案依赖用户权限、知识库版本、Prompt 版本和上下文，如果这些条件不同，缓存答案可能泄露数据或返回过期内容。更稳的缓存对象是 embedding、静态配置、热点检索结果，并在 key 里带上租户和版本信息。

数据库、缓存、性能和部署：标准答案
  1. Q：关系型数据库和 NoSQL 有什么区别？

    答：关系型数据库强调结构化 schema、事务和 SQL 查询，适合强一致业务数据；NoSQL 更灵活，适合文档、键值、图或高扩展场景。AI 应用里常见组合是关系库保存业务状态，向量库保存 embedding。

  2. Q：缓存适合缓存什么？

    答：可以缓存 embedding、热点检索结果、静态配置和部分常见问答，但要注意数据更新和权限隔离。不能盲目缓存所有 LLM 输出，因为用户上下文、权限和知识库版本不同会导致答案不一致。

  3. Q：如何降低 LLM 应用延迟？

    答：可以减少上下文 token、用更快模型、流式输出、缓存 embedding 或热点结果、并发检索和生成前处理、批量 embedding、优化向量库索引。定位时先看 retrieval、rerank、LLM generation 哪段耗时最高。

  4. Q：如何降低 LLM 应用成本？

    答：减少无用上下文、控制输出长度、缓存稳定结果、选择合适模型、复用 embedding、离线批处理评估。成本优化不能牺牲关键质量，要用评估集验证改动没有明显退化。

  5. Q：如果 QPS 提升 10 倍，瓶颈可能在哪里？

    答：可能在 LLM API 限流、向量库检索、embedding 服务、PDF 入库、网络连接池、同步阻塞代码和日志写入。排查要看 metrics：请求量、延迟分布、错误率、外部服务耗时和资源使用。

  6. Q：为什么用环境变量管理配置？

    答：模型地址、API key、数据库 URL 和 collection 名称在不同环境会变化，而且密钥不能写死进代码。环境变量让配置和代码分离，也方便部署和密钥管理。

  7. Q：Docker Compose 在本地开发中解决什么问题？

    答：它可以把 Qdrant 这类依赖服务用统一配置启动，减少手动安装和环境差异。对实习项目来说，它能证明你的项目不是只在笔记本里临时跑，而是有基本可复现运行环境。

  8. Q：CI/CD 至少应该跑哪些检查？

    答：至少跑单元测试、格式检查、静态检查和关键脚本 smoke test。AI 应用还可以跑小规模 golden set，防止 Prompt、检索参数或接口改动导致质量回归。

安全、合规和 Prompt Injection：标准答案
  1. Q：RAG 系统为什么容易受 prompt injection 影响？

    答：因为 RAG 会把检索到的文档内容放进模型上下文，如果文档里有恶意指令，模型可能把它当成高优先级指令执行。防御要区分“文档是数据，不是指令”，并结合权限控制、输出检测和工具审批。

  2. Q：如何防止越权检索？

    答：检索前必须基于用户身份、租户和知识库权限做强制过滤，不能只在 Prompt 里要求模型不要引用无权限文档。metadata filter 是基础，但复杂场景还需要权限服务和审计。

  3. Q：工具调用为什么需要权限控制？

    答：工具可能执行真实动作，例如发邮件、删数据、查订单。模型不能拥有无限权限，高风险工具需要参数校验、权限检查、人工确认和审计日志。Agent 安全的核心是限制工具能力边界。

  4. Q：日志里能不能记录完整用户问题和回答？

    答：要谨慎。完整问题和回答可能包含隐私、商业数据或密钥。生产环境应做脱敏、最小化记录、访问控制和保留周期管理。调试价值和隐私风险要平衡。

  5. Q：如果模型生成不安全内容，系统怎么处理？

    答：可以在输入和输出两侧加入安全检测，对高风险类别拒答或降级；对工具调用增加审批；对检索内容做权限和敏感信息过滤。不要完全依赖模型自己判断安全性。

系统设计场景题：标准答案
  1. Q：设计企业内部知识库问答系统。

    答：先明确目标是基于企业文档回答并带引用。架构上包括文档上传、异步解析切分、embedding 入库、向量检索、rerank、Prompt 生成、sources/trace 返回、评估和权限控制。风险点是越权检索、文档更新、检索不准、延迟和幻觉。

  2. Q：设计客服机器人，如何接订单查询、退款、知识库和人工转接？

    答：可以用 Agent 编排：知识性问题走 RAG，订单类问题调用订单工具，退款类高风险操作需要权限和人工确认，无法解决时转人工。核心是工具 schema、状态管理、权限控制、trace 和失败降级。

  3. Q：用户上传 1 万份文档，如何设计入库、索引、检索和更新？

    答：上传后进入异步任务队列，分批解析、切分、embedding、写入向量库；使用稳定 document_id/chunk_id 支持幂等和增量更新；索引时保存 metadata；检索时用权限 filter、向量召回、rerank 和缓存。还要有任务状态、失败重试和监控。

  4. Q：回答延迟从 3 秒变成 20 秒，怎么定位？

    答：先看 trace 拆分耗时：请求排队、检索、rerank、Prompt 构造、LLM 首 token、LLM 总生成、网络。再看外部服务状态、token 数、top_k、模型输出长度和错误重试。不要直接猜是模型慢。

  5. Q：用户反馈答案胡说，怎么排查？

    答：先看 sources 是否相关；如果 sources 不相关，是检索问题；如果 sources 相关但答案不支持，是生成或 Prompt 问题；如果 sources 不完整，是 chunking/top_k/rerank 问题；最后把失败样本加入 golden set 做回归。

  6. Q：支持多用户多知识库，如何做权限隔离？

    答：数据层记录 tenant/workspace/user metadata，检索时强制 filter；API 层做认证授权；日志和缓存也要按租户隔离；高安全场景可以按租户拆 collection。不能只在前端隐藏知识库。

  7. Q：每个答案必须有引用，系统怎么保证？

    答：Prompt 要强制 citation 格式，context 要带编号，后处理要校验答案是否包含引用，LLM-as-Judge 或规则校验要检查引用是否对应 sources。若没有足够证据，应返回 fallback，而不是硬生成无引用答案。

  2. Q：和网上教程项目相比，你的差异在哪里？

    标准答案：差异不在于用了某个罕见框架，而在于我把链路做成了可解释和可评估的结构。比如回答不只返回 answer，还返回 sources 和 trace；Prompt 不只是写死一个版本，而是支持版本切换和 A/B 报告；Agent 不是口头概念，而是有 planner、executor、tool result 和失败重试。边界是检索质量和生产化能力还需要继续增强。

    标准答案：目前它主要是学习和作品集项目，没有真实生产用户，所以我不会用真实线上指标包装它。它的价值主要通过代码结构、测试、评估脚本和实验报告证明：能跑通完整 RAG/Agent 链路，也能说明我理解 AI 应用从 ingestion 到 retrieval、generation、evaluation 的工程闭环。后续如果接真实用户，才需要补日志、权限、监控和反馈闭环。

生产级边界
  1. Q：你说这是生产级 RAG 吗？

    标准答案：我不会说它是生产级 RAG。它现在是一个工程化学习基线，已经有核心链路、接口、trace 和评估，但生产级还需要权限系统、异步任务队列、监控告警、SLO、数据隔离、Prompt injection 防护、灰度发布和更大规模评估。面试中我会把它定位为“可继续生产化演进的基线”。

  2. Q：如果上线给公司内部用，最大风险是什么？

    标准答案：最大风险有三类：第一是权限风险，用户可能检索到不该看的文档；第二是质量风险，检索不相关但 LLM 仍然生成看似可信的答案；第三是稳定性风险，LLM、embedding 或 Qdrant 外部依赖超时失败。所以上线前要补认证授权、metadata filter、低置信 fallback、日志监控和超时重试策略。

  3. Q：为什么现在没有做用户系统和权限？

    标准答案：因为当前项目目标是先把 RAG 和 Agent 的核心工程链路做清楚，用户系统和权限是生产化阶段的必要模块，但不是验证 RAG 基线的最小路径。我没有忽略这个问题，后续多知识库或多用户场景必须加 tenant/user metadata、检索过滤和接口鉴权。

  4. Q：为什么没有做异步任务队列？PDF 入库不慢吗？

    标准答案：当前实现适合小规模本地学习和演示，入库直接由接口或脚本触发。真实场景下 PDF 解析、chunking、embedding 和写库都可能耗时，应该异步化成 job：上传后返回 job id，后台处理并提供状态查询。这个是我后续生产化改造的优先方向之一。

检索质量边界
  1. Q：你怎么保证检索到的内容一定相关？

    标准答案：没有系统能保证“一定相关”，但我做了几层处理：向量检索高召回初筛，用 sources 和 trace 暴露证据，用离线 eval 检查检索质量。我还专门做过一次检索策略对比——想用 MMR 提升多样性，结果实测 MRR 反而回退，所以据数据保留了 similarity。后来又实测了 BM25 + 向量的 hybrid（含 code-aware 分词修复），全集上 dense 仍占优、仅代码符号难例略胜，默认也没换。残留边界是 rerank / score-threshold 还没做——这是后续方向。

  2. Q：documents 不为空但内容错了怎么办？

    标准答案：这是比空检索更危险的情况，因为模型会基于错误上下文生成看似合理的回答。我现在的处理是：生成层强制只基于 sources 回答并暴露 trace，评估层用 LLM-as-Judge 的 groundedness 维度抓「不被证据支撑」的回答、把失败样本回灌 golden set 做回归。检索层的 score-threshold 过滤和排序层的 rerank 是还没做的后续方向 🔨。关键是把「低质量检索」当成一个独立于「空检索」的问题来治理。

  3. Q：你做过检索排序优化吗？效果如何？✅

    标准答案：我做的是检索策略对比实验（不是 rerank）：在 27 条 golden set（含 3 条实测难例）上对比 similarity 与 MMR，定义 source hit rate / MRR / 来源覆盖率指标。结果 MMR 每个 λ 下 MRR 都回退（similarity 0.901 vs MMR 0.892@λ0.3），它没提升任何 case 的 rank、只把两个 hard case 排得更后，所以我据数据保留 similarity、把 MMR 设为可配置 opt-in，before/after 写进 experiments/ 报告。如果要上 rerank（cross-encoder），我会用同一套 before/after 方法验证——但目前没做，不会硬说做了。这就是「有数据就敢否决自己的改动」。

  4. Q：你做了 hybrid search 吗？效果如何？✅

    标准答案：做了，而且有完整实验。我实现了 BM25 + 向量的 hybrid（EnsembleRetriever / RRF 融合，BM25 语料从 Qdrant 回读、保证和向量索引一致）。过程中发现 BM25 默认按空格分词，对 configure_optimizers(self): 这类代码符号零召回，于是实现 code-aware 分词器（按非字母数字切分），修复后难例进 Top-2。评测结论：全集 27 条上 dense 仍占优（MRR 0.901 vs 最佳 hybrid 0.873），仅 8 条代码符号难例子集上 hybrid 以 0.3/0.7 权重略胜（0.562 vs 0.525）。所以默认保留 similarity、hybrid 作为 opt-in——这是我第二个「用数据否决预期」的实验。

  5. Q：你的 top_k=7 是否有充分依据？

    标准答案：目前只能说是基于小规模 golden cases 的经验默认值，不能说是全局最优。top_k 本质是召回、噪声、成本和延迟的权衡。更严谨的做法是扩大评估集，对比 top_k=3/5/7/10 的 Recall@K、groundedness 和 latency，然后再确定默认值。

  6. Q：chunk size=800、overlap=100 是怎么来的？

    标准答案：这是当前阶段的经验参数，目的是让 chunk 保留较完整语义，同时避免上下文过长。它不是最优证明。后续应该通过 chunk size 对比实验，比如 400/800/1200，观察 retrieval hit rate、上下文噪声和回答质量，再做更有证据的选择。

评估边界
  1. Q：你的评估集只有 27 个 case，能说明什么？

    标准答案：27 条 case（含 3 条实测 hard case）能说明我建立了有区分度的评估闭环——hard case 正是通过约 20 次 probing 实测筛出来的，不是凭感觉编的。但它还不能说明系统在大规模问题上稳定可靠，我把它作为 smoke test 和回归基线，而不是最终质量证明。下一步扩展更多问题类型：无答案问题、长文档问题和更多容易混淆的难例。

  2. Q：LLM-as-Judge 可靠吗？会不会自己也错？

    标准答案：LLM-as-Judge 不是绝对可靠，它可能有偏置、不稳定、被答案格式影响，甚至使用外部知识。它的价值是低成本批量提供质量信号，适合回归测试；严肃评估还需要固定 judge prompt、结构化 schema、人工抽检和失败样本分析。

  3. Q：Judge 使用本次 answer 的 sources，会不会不够独立？

    标准答案：会，这是当前评估边界。它能评估“答案是否被本次返回的 sources 支撑”，但不能判断知识库里是否存在更完整的证据。更强的方案是引入独立 judge retriever 或人工标注 expected sources，把 answer sources 和 judge sources 分开。

  4. Q：Prompt A/B 只有小样本，能证明 v1/v2 谁更好吗？

    标准答案：不能证明全局谁更好，只能说明在当前固定样本和配置下的观察结果。Prompt A/B 的价值是建立可对比流程：固定问题集、检索配置、模型和 judge，只改变 Prompt。后续需要扩大样本并加入人工复核，才能做更稳的结论。

Agent 边界
  1. Q：你的 Agent 是真正的 Agent 吗，还是 if-else？

    标准答案：工具选择是模型通过 function calling 做的 ✅：我把工具的 input_schema 作为 tool parameters 喂给模型，让它按自然语言意图选工具、填参，规则式 planner 只作 fallback ✅。它有工具注册、executor、ToolResult、结构化失败记录和 Agent trace。在此之上是多轮 agent loop ✅：每轮模型决定调工具还是直接作答，工具结果以 ToolMessage 回灌供下一轮决策，最多 4 轮、超预算强制收敛——这是 action→observation 的自主循环，不是 if-else。我诚实的边界是：无跨会话长期 memory、每轮只执行一个工具调用。

  2. Q：你既用 LLM function calling 又留规则式 planner，为什么是双层？

    标准答案：function calling 让模型对自然语言意图泛化得更好，是主路径 ✅；但我保留规则式 planner 作 fallback：模型不支持 tool calling、超时、或返回非法工具名时兜底，既省 token 又保证可控可测。核心判断是「什么时候让 LLM 选、什么时候规则更稳」，不是非此即彼。多轮决策已由 agent loop 承接 ✅，再往上才是 LangGraph 状态流。

  3. Q：summary_tool 是真摘要吗？

    标准答案：是真 LLM 摘要 ✅（带长度控制和空输入/空输出兜底），替换了最初那个字符串截断的占位实现。诚实说：它早期确实是确定性截断，先用来验证工具注册/选择/执行闭环；升级成 LLM 摘要后，Agent 才有了「检索 / 拆解 / 摘要 / 兜底」多类真实工具。

  4. Q：你的 Agent 没有 memory，有什么影响？

    标准答案：要分两层说：单次 run 内，agent loop 用消息历史维护多轮工具调用的上下文，模型能基于上一轮结果继续决策 ✅；跨会话的长期 memory 没做，意味着它记不住用户偏好和历史任务——这是有意取舍，当前聚焦单次 run 的可控执行。后续如果加长期 memory，需要设计 session state、历史压缩、隐私边界和清理策略。

  5. Q：工具失败重试会不会造成重复副作用？

    标准答案：会，所以不是所有工具都适合自动重试。当前 retry 主要用于 retrieval_tool 这类读操作，副作用低；如果是写数据库、支付、发邮件等工具，就必须考虑幂等 key、人工确认和错误分类。这个边界在真实 Agent 里非常重要。

  1. Q：你的项目如何防 prompt injection？

    标准答案：当前还没有完整防护，只在 Prompt 中要求基于 context 回答，这不够。真实防护需要把检索文档视为数据而不是指令，加输入输出安全检测、工具权限控制、敏感操作审批和攻击样本评估。这个是生产化前必须补的边界。

11. 项目拷打清单：RAG 子项目

拷打点 1：/ask 链路

面试官问：完整讲一下 /ask 请求怎么走。

答题骨架：router 接收 AskRequest，调用 ask_question()；service 里先 analyze_query()，再 plan_retrieval()，再 get_retriever().invoke() 检索，之后 format_context() 拼上下文，最后 Prompt + LLM 生成，返回 answer、sources、trace。

容易被追问：为什么空问题直接 fallback？为什么 format_context() 是格式化 documents，不是格式化模型输出？

拷打点 2：sources

作用：记录本次回答引用到的文档片段。

工程价值：让用户能溯源，也方便你判断错误来自检索还是生成。

当前边界：sources 只能说明模型拿到了这些片段，不代表片段完整、排序最优、一定支持最终结论。

拷打点 3：trace + 结构化日志

作用：trace 记录 query analysis、retrieval planning、retrieval、generate answer；结构化 logging ✅ 记录请求耗时与外部调用错误。

工程价值：trace 是请求内业务轨迹、logging 是跨请求运维信号——答案错了能拆开看是问题分析、检索、生成还是外部服务的问题。

当前边界：已有标准库 logging ✅，但还不是 metrics + distributed tracing；request-id 关联是后续增强。

拷打点 4：top_k=7 与检索策略

推荐说法：top_k 是召回率、噪声、token 成本和延迟之间的折中，基于 golden cases 选 7 作为经验默认。我还做过一次有数据的检索策略对比 ✅。

真实结论（王牌）：在 27 条 golden set（含 3 条难例）上对比 similarity 与 MMR，MMR 每个 λ 下 MRR 都回退（similarity 0.901 vs MMR 0.892@λ0.3），唯一收益是多样性且 judge 无提升，所以默认保留 similarity、MMR 设为可配置 opt-in——有数据就敢否决自己的改动。

不要硬吹：不能说 7 是全局最优；rerank / score-threshold 仍是没做的待补方向 🔨（hybrid 已实测 ✅，负结果、保留 similarity）。

拷打点 5：比较类问题

当前实现：comparison_retrieval 会把不同 source 的 documents 交错排列，避免证据都来自单一来源。

工程价值：比较类问题通常需要多来源证据，交错排列是一种很轻量的多来源覆盖策略。

当前边界：它不是严格的 diversity rerank，也没有判断每个 source 是否真的覆盖比较对象。

拷打点 6：文档入库

链路：/documents/upload 保存文件，/documents/ingest 调 ingest_file()，Markdown/PDF 分别 load、chunk、ingest_chunks() 写入 Qdrant。

加分点：上传接口限制 .md 和 .pdf，并通过 Path(filename).name 降低路径穿越风险。

当前边界：没有文件大小限制、内容安全扫描、异步任务队列，也没有上传后自动增量索引调度。

拷打点 7：Prompt A/B

当前实现：通过 QA_PROMPT_VERSION 切换 qa_prompt_v1 和 qa_prompt_v2，用固定 golden questions + judge 生成报告。

工程价值：不是凭感觉改 Prompt，而是让 Prompt 版本变化可比较、可回归。

当前边界：样本只有小规模 golden cases，不能外推成完整 Prompt 平台或大规模实验。

拷打点 8：LLM-as-Judge（王牌，主动打出去）

最强设计点：judge schema 用 Pydantic model_validator 强制 overall_pass 由 relevance / completeness / groundedness / format 四维分数推导，模型不能自己说「我通过了」——从机制上杜绝评判模型自评注水。这是「怎么评 LLM 输出」最稀缺的判断，先讲它。

当前实现：judge_answer() 把 question、answer、sources 交给 judge prompt，strip 掉 markdown code fence 后 model_validate_json 解析并校验分数。

工程价值：结构化评估四维 + A/B + 报告留档；最近一轮 27 条 golden set 27/27 通过。

当前边界（诚实）：Judge 使用本次 answer 的 sources、不是独立证据检索；judge 模型本身也可能偏置或不稳定，所以只用作回归信号、不当绝对真值。

拷打点 9：流式问答

当前实现：/ask/stream 返回 application/x-ndjson，事件包括 answer_delta、sources、trace、done。

工程价值：减少用户等待感，前端可以边接收边展示。

当前边界：检索阶段仍然要先完成；流式主要覆盖生成阶段。

拷打点 10：当前 RAG 的真实边界

已落地 ✅：结构化 logging、客户端复用、流式统一重试、Docker + pydantic-settings、扩到 27 条的 golden set，以及一次 similarity vs MMR 的检索策略对比实验。

诚实说没做 🔨：rerank / cross-encoder 精排、score-threshold 过滤还没做；hybrid 已实现并实测 ✅——全集 dense 仍占优（MRR 0.901 vs 0.873），仅代码符号难例 hybrid 略胜，所以保留 similarity 默认、hybrid opt-in；query analyzer 仍是规则式。我把它定位为「可验证的 RAG 工程基线」，不是生产级 RAG。

12. 项目拷打清单：Agent 子项目（最终版）

叙事翻转：这一章已从「轻量规则式编排」升级为「多轮 agent loop」：function calling 选工具、工具结果回灌、步数预算、强制收敛全部落地 ✅、现在就能理直气壮讲（web search 已决策不做，别提）。核心取舍：用 function calling 让模型对意图泛化、用循环让它基于工具结果多步决策、用规则式 planner 和步数预算保证可控——这套设计本身就是面试加分点。

拷打点 1：Agent 链路

链路：/agent/run → run_agent_loop()：system + 用户消息起步 → 每轮 bind_tools 调用模型，模型决定发 tool_call 还是直接作答 → executor 执行工具 → 结果裁剪成紧凑 JSON 以 ToolMessage 回灌 → 下一轮决策；最多 4 轮，超预算基于已有上下文强制生成最终答案。返回 answer、sources（跨轮聚合）、steps、termination_reason。

定位：✅ 多轮 tool-calling 循环（ReAct 式 action→observation），function calling 选工具填参，规则式 planner 作 fallback ✅。边界：无跨会话 memory、每轮只执行第一个 tool call——这是有意的取舍，不是缺陷。

拷打点 2：planner 为什么是「function calling + 规则式 fallback」双层

推荐说法 ✅：工具的 input_schema（registry 里已定义 ✅）直接作为 function/tool parameters 喂给模型，让模型按自然语言意图选工具、填参——比关键词 if/else 泛化好得多。但我保留规则式 planner 作 fallback：模型不支持 tool calling、超时、或返回非法工具时兜底，既省 token 又可控可测。

取舍亮点：什么时候让 LLM 选、什么时候规则更稳——这正是面试官想听的判断力。

拷打点 3：工具清单（多工具）

  • retrieval_tool ✅：调用 RAG ask_question()，带 grounded answer + sources。
  • question_decompose_tool ✅：对比较/多子问题做规则式拆解，逐个检索后聚合 sub_results。
  • summary_tool ✅：真 LLM 摘要（空输入 / 空输出兜底 + 长度上限），已替换原来的字符串截断占位。
  • fallback_tool ✅：空问题或无需检索时兜底。

诚实边界：web search 已评估决定不做（代码无、简历无），不要提「集成网络搜索」；工具链就是检索 / 摘要 / 拆解 / 兜底这 4 个，讲深比讲多更有用。

拷打点 4：失败处理与 retry

当前实现 ✅：向量检索的有限 retry 已下沉到 RAG 服务层（贴近真实失败点），Agent 层不再做粗粒度整体 3 次 retry——避免多子问题场景放大下游调用。Agent 层把每次 attempt 的 status、error_type、error 结构化记录进 ToolResult。

讲取舍：“retry 要靠近失败点”——向量库失败在检索层、生成失败在 generation 层，Agent 只编排和记录。这比「所有工具统一 retry 3 次」专业得多。

边界：暂无指数退避、按错误类型分级的 retry policy；这是有意的简化，够用且能讲清。

拷打点 5：Agent trace

记录 ✅：analyze_question、plan_tool、execute_tool，每步带 status/detail；loop 路径另记每轮 steps（round、tool_name、tool_args、tool_status）与 termination_reason。

已落地 ✅：plan_tool 这步记「模型选了哪个工具、填了什么参数」，让工具选择可复盘——这是 function calling 落地后最有说服力的证据。

和 RAG trace 区别：Agent trace 记上层决策；RAG trace 记工具内部的检索与生成。

拷打点 6：agent loop 设计细节（被问到就是送分题）

为什么每轮只执行第一个 tool call：简化执行模型、保证 trace 线性可复盘，其余 tool_call 回灌「skipped」消息明确告知模型；并行工具执行是后续优化。为什么 max_steps=4：当前工具链下多数任务 1-2 轮可完成，4 是能力和成本的折中；超预算不报错，而是基于已有上下文强制生成最终答案（termination_reason 区分 final_answer / max_steps / failed）。token 怎么控：工具结果回灌前裁剪成 status + answer + 来源标签的紧凑 JSON，不回灌检索原文全文。工具失败怎么办：异常捕获成结构化失败结果回灌给模型，让它自己决定换工具还是直接作答，循环不中断。

正确定位：“它是一个 LLM 驱动、步数受控、失败可兜底、全程可 trace 的多轮工具循环；跨会话 memory 和 LangGraph 持久化是明确的下一步。”诚实划这条线，比含糊吹「自主 Agent」更得分。

拷打点 7：下一步怎么升级（agent loop 做完后）

  • LangGraph：持久化状态、checkpoint、可恢复执行、human-in-the-loop。
  • 长期 memory：跨会话 session state、历史压缩、隐私边界与清理策略。
  • Trace eval：对「工具选择是否正确、执行路径是否最优」做离线评估。
  • 工具治理：权限、超时预算、敏感操作 guardrails 与审批。
  • 成本/延迟：并行执行相互独立的工具、轮内多 tool-call 支持。

拷打点 8：Agent 和 RAG 的关系

推荐说法：RAG 是 Agent 的核心工具之一（retrieval / decompose 都建在它上面）；Agent 层负责理解意图、选工具、编排执行、记录上层 trace，RAG 层专注检索与 grounded 生成。把请求路由从 RAG 接口拆到 Agent 层，让 RAG 接口聚焦问答、Agent 层聚焦编排——这是关注点分离。

加分边界：工具数量仍有限，复杂任务分解和多步推理是后续；当前价值在「LLM 选工具 + 可控 fallback + 全链路可 trace」这条干净闭环。

14. 背诵模板：直接练这些

模板 1：30 秒讲 /ask 链路

/ask 请求进入 FastAPI 的 ask router 后，会被解析成 AskRequest，然后交给 service 层的 ask_question()。service 会先用 analyze_query() 分析问题类型和是否需要检索，再用 plan_retrieval() 决定检索策略、检索 query 和 top_k。如果需要检索，就通过基于 Qdrant 的 LangChain retriever 做相似度检索，拿到相关 documents。之后 format_context() 把文档片段组织成 LLM 输入的上下文，再结合 Prompt 和 LLM 生成答案。最后返回 answer、sources 和 trace，其中 sources 用于答案溯源，trace 用于调试和评估 RAG 链路。

模板 2：为什么要拆模块

我把链路拆成 query analyzer、retrieval planner、retriever、generation，不是为了复杂化，而是为了职责清楚和可测试。query analyzer 只负责理解问题类型；retrieval planner 只负责决定怎么检索；retriever 只负责访问向量库；generation 只负责 Prompt 和 LLM 调用。这样后续要加 rerank、换向量库、改 Prompt、做评估时，不需要改一个巨大的函数。

模板 3：为什么要有 sources

sources 的作用是记录本次回答引用到的文档片段。它的工程价值是让用户能追溯答案依据，降低 RAG 回答的黑盒感，也方便我调试答案错误到底是检索问题还是生成问题。当前边界是，sources 只能说明模型拿到了这些片段，不代表这些片段一定完整、排序最优或完全支持最终答案。

模板 4：为什么要有 trace

trace 的作用是记录一次请求内部发生了什么，比如问题分析、检索规划、检索结果数量和生成状态。工程价值是可观测性：如果答案不好，我可以看是 query analysis 错了、retrieval 没召回，还是 generation 出错。当前边界是它还只是轻量 trace，没有接入正式日志、metrics 或分布式 tracing。

模板 5：top_k 为什么选 7

top_k=7 是当前项目基于小规模 golden cases 的经验默认值，本质是在召回率、噪声、token 成本和延迟之间折中。k 太小可能漏掉关键证据，k 太大会引入无关片段，导致 LLM 上下文变脏、成本和延迟增加。所以我不会说 7 是全局最优，后续应该随着语料和评估集扩大继续调参。

模板 6：你的 Agent 是什么

我的 Agent 子项目是建立在 RAG 之上的多轮工具循环。模型每轮通过 function calling 从工具 registry 中选工具并填参（规则式 planner 作 fallback），executor 执行后把结果裁剪回灌给模型，由它决定继续调工具还是给出最终答案；最多 4 轮、超预算强制收敛，全程记录 steps 和 termination_reason。我诚实的边界是：没有跨会话长期 memory、每轮只执行一个工具；定位是一个 LLM 驱动、步数受控、规则可兜底、全程可 trace 的多轮编排闭环。

模板 7：当前项目为什么还不是生产级

当前项目已经有 RAG 闭环、sources、trace、结构化日志、Prompt 版本、离线评估，以及多轮 agent loop ✅，但我不会把它说成生产级系统。主要边界是：检索默认仍是 similarity（MMR、hybrid 都实测过、数据上没跑赢才保留的），rerank 和相关性阈值还没做；评估集规模仍偏小；可观测有 logging 但还没 metrics/分布式 tracing；Agent 无跨会话 memory、无持久化状态。后续优先补 rerank、扩样本、指标监控和 LangGraph 持久化。

“项目已经生产可用。” 缺少监控、安全、规模化、权限、队列、SLO 等生产要素。 “这是可验证的工程基线，具备继续生产化演进的基础。”

P1 rerank 公开 RAG 面试题经常问 rerank，它能体现你知道向量召回和最终排序不是一回事。 先做 cross-encoder 或 lightweight reranker 的可插拔接口；不用一上来做复杂平台。 “第一阶段用向量库召回，第二阶段用 reranker 对候选片段重排。”

P1 observability 指标 AI 应用岗位越来越重视 eval 和可观测，不只是能跑。 在 trace 或日志里记录 retrieval duration、generation duration、document_count、fallback_rate、prompt_version。 “答案错了我能定位是检索、Prompt、LLM 还是外部服务问题。”
P1 Agent 工具契约 当前 tool 只是 Python 函数，面试官会追问 tool schema、参数校验和错误处理。 给每个 tool 定义输入输出 Pydantic schema，统一 ToolResult，错误分类。 “工具不是随便调函数，而是有稳定输入输出契约。”
P2 后台 ingestion 任务 文档上传和向量化可能耗时，真实应用不应该阻塞 HTTP 请求太久。 先做任务状态表或简单 job id；后续可接 Celery/RQ。 “上传、解析、embedding、入库应该异步化，避免长请求阻塞。”
P2 安全和权限边界 AI 应用常见风险是越权检索、prompt injection、敏感信息泄露。 先加文件大小限制、content type 校验、metadata filter 预留、Prompt injection 测试 case。 “RAG 安全不只看模型，还要控制检索范围和工具权限。”

版本 A：检索评估增强

目标：把“top_k=7 是经验选择”升级成“有评估证据的选择”。

版本 B：相关性过滤 + fallback

目标：解决“检索到了但不相关”的核心风险。

  • 让 retriever 返回 score，设置低置信阈值。
  • 低于阈值时不进入 LLM，返回 fallback。
  • trace 里记录 top score 和过滤原因。

简历表达：“为 RAG 链路加入低置信检索保护，减少无关上下文导致的幻觉风险。”

版本 C：Agent 工具 schema

目标：让轻量 Agent 更像一个工程化编排层。

  • 定义 ToolInput、ToolOutput、ToolError schema。
  • 把 retrieval、summary、fallback 的输出统一起来。
  • 在 trace 中记录 selected_tool、input shape、status、attempts。

简历表达：“设计工具注册和结构化执行结果，支持工具失败重试和 Agent trace。”

版本 D：可观测和 benchmark

目标：把“能跑”升级成“能测、能排查”。

  • 记录 retrieval/generation/total latency。
  • 统计 fallback rate、empty retrieval rate、judge pass rate。
  • 生成一次 benchmark 报告，不要夸大成稳定生产指标。

简历表达：“加入 RAG 请求 trace 和延迟 benchmark，用于定位性能瓶颈和质量回归。”

/ask 链路 /ask 先进入 FastAPI router，解析成 AskRequest，再进入 service 层完成问题分析、检索规划、向量检索、上下文格式化和 LLM 生成。
sources sources 的作用是让答案能回到原始文档片段，方便用户和开发者判断答案依据，但它不能证明答案一定正确。
trace trace 是为了调试和评估链路，帮助定位问题出在 query 分析、检索、Prompt 还是生成阶段。
top_k top_k 我没有拍脑袋选，而是用 golden cases 做过评估，选择了命中效果和上下文长度之间更平衡的值。
RAG 边界 RAG 用离线 eval 对比过 similarity、MMR 和 BM25+向量 hybrid ✅，两轮实验都没跑赢基线、据数据保留 similarity；真实边界是 rerank / score-threshold 还没做，排序精度是下一步。
Agent 定位 我的 Agent 是一个多轮工具循环：模型每轮决定调工具还是作答、工具结果回灌再决策，步数预算防失控；没有长期 memory，不吹「全自主」。
生产化边界 这个项目现在是学习和实习展示级工程基线，生产化还需要权限、限流、监控、异步任务、成本控制和更完整评估。
AI 辅助开发 代码有 AI 辅助，但我通过逐模块复盘、写测试、做评估和能定位到函数级别来证明自己理解链路，而不是只会复制。
下一步优化 如果继续做两周，我会优先补 retrieval eval、rerank 或相关性过滤、trace 指标和更大的 golden set，而不是先堆复杂功能。

19.3 现场 Coding 最小稳态流程

Coding 面不要求你像算法竞赛选手，但你必须体现“能思考、能写、能测”。按照下面 7 步走，比直接闷头写更稳。
  1. 复述题意：确认输入、输出、是否有重复、是否有序、边界规模。
  2. 举例：拿一个正常例子和一个边界例子说明理解。
  3. 先给暴力：如果想不到最优，先说暴力思路和复杂度。
  4. 优化：常见方向是哈希表、双指针、滑动窗口、栈、二分、BFS/DFS。
  5. 写代码：变量名清楚，边界先处理，不追求炫技。
  6. 跑样例：口头 trace 一遍主要例子。
  7. 讲复杂度：明确时间复杂度和空间复杂度。
题型 优先模板 你至少要会的题
哈希表 用 dict/set 存 seen 或计数。 两数之和、最长连续序列的简化版、字符频次比较。
栈 遇到右括号检查栈顶。 有效括号、最小栈思路。
双指针 左右指针向中间移动或快慢指针。 反转字符串、合并有序数组、删除有序数组重复项。
滑动窗口 右扩张、左收缩、维护窗口状态。 无重复字符最长子串、长度最小子数组。
二分 维护 left/right/mid 和循环不变量。 二分查找、搜索插入位置。
BFS/DFS 队列或递归，注意 visited。 岛屿数量简化版、二叉树层序遍历。
