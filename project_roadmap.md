# 技术文档 / API 文档智能问答助手 — 项目路线图

> 基于 RAG 技术栈构建的技术文档问答系统，核心目标是**帮助开发者快速从技术文档中获取准确、可信、带引用的回答**。

---

## 项目定位

**"官方文档检索 + 可信回答 + 引用来源"**

不是通用聊天机器人，而是一个能理解文档结构、区分版本差异、精确引用来源的技术文档问答助手。

---

## 技术栈

| 组件 | 选型 |
|------|------|
| LLM | Kimi (OpenAI-compatible API) |
| Embedding | bge-m3 (Ollama) |
| 向量数据库 | Qdrant (Docker) |
| 框架 | LangChain |
| 后端 | FastAPI |
| 语言 | Python 3.14 |

---

## 核心场景（贯穿 V1 ~ V3）

### 场景 1：参数 / 字段解释

- `temperature` 是什么意思
- `messages` 是否必填
- `top_p` 和 `temperature` 有什么区别

**要求**：定义准确、引用准确、不能胡编默认值

### 场景 2：接口调用方式

- 这个接口的最小请求示例是什么
- curl 怎么写
- Python SDK 怎么写

**要求**：命中接口定义、结合请求示例、回答结构化

### 场景 3：错误码 / 报错排查

- 为什么会报 401
- 429 是什么意思
- 参数错误通常出在哪里

**要求**：命中错误说明、给排查步骤、区分"文档明确说明"和"经验性推断"

### 场景 4：版本差异 / 迁移

- 这个字段新版本还在吗
- v1 到 v2 有什么变化
- 老接口是不是废弃了

**要求**：强依赖版本元数据、标注版本、检索不能混不同版本内容

## 系统架构（6 层）

```
┌──────────────────────────────────────────────┐
│  6. 评估层 — 测试集 + 自动化指标             │
├──────────────────────────────────────────────┤
│  5. 交互层 — 单轮问答 / 追问 / 过滤指定      │
├──────────────────────────────────────────────┤
│  4. 回答层 — 结论 + 依据 + 引用 + 保守表达   │
├──────────────────────────────────────────────┤
│  3. 检索层 — 向量检索 → 混合检索 → 过滤+重排 │
├──────────────────────────────────────────────┤
│  2. 结构化层 — 元数据提取 (版本/模块/接口等)  │
├──────────────────────────────────────────────┤
│  1. 文档层 — 多格式解析 (MD/HTML/OpenAPI)     │
└──────────────────────────────────────────────┘
```

## V1：能查准的文档问答助手

> **核心命题**：用户问文档问题时，能不能稳定找到正确内容并回答出来。

### V1 目标能力

- [ ] 支持导入技术文档（Markdown）
- [ ] 支持导入 API 文档（OpenAPI / Swagger）
- [ ] 用户输入问题后能检索相关片段
- [ ] 回答里带出处引用
- [ ] 上下文不够时明确说不知道

### V1 各层实现要点

#### 1. 文档层

- 支持 Markdown 技术文档导入
- 支持 OpenAPI / Swagger YAML/JSON 导入
- 明确主要文档来源，先做深再做广

#### 2. 结构化层

- Markdown：按标题层级（`#` `##` `###`）切分，保留章节路径作为元数据
- OpenAPI：按 endpoint 拆分，每个 API 路径为一个 chunk
- 代码块完整性：检测 ` ``` ` 边界，避免将代码块切断
- Chunk 元数据：`source`、`section_path`、`doc_type`、`has_code`

#### 3. 检索层

- 基础向量检索（当前已有）
- 调优 `chunk_size` 和 `top_k` 参数

#### 4. 回答层

- 技术文档专用 System Prompt
- 回答必须包含：直接结论 + 关键依据 + 引用出处
- 上下文不足时返回保守表达（已有 `FALLBACK_ANSWER`）

#### 5. 交互层

- FastAPI 服务（`POST /ask`、`POST /ingest`、`GET /health`）
- 单轮问答即可

#### 6. 评估层

- 准备一批覆盖 4 个场景的测试问题
- 人工观察：是否检索到正确片段、回答是否引用错、是否胡编

### V1 明确不做

- ~~多轮复杂 Agent~~
- ~~自动调用真实 API~~
- ~~自动修复代码~~
- ~~工具调用编排~~
- ~~超复杂工作流~~

## V2：更像技术助手的能力

> **核心命题**：不是"能不能答"，而是"答得像不像一个技术支持工程师"。

### V2 新增能力

- [ ] 按产品 / 版本 / 文档类型过滤检索
- [ ] 支持 HTML 技术站点文档
- [ ] 混合检索（向量 + BM25 关键词）
- [ ] Rerank 重排序（Cross-Encoder）
- [ ] 回答自动区分类型：概念解释 / 参数说明 / 错误排查 / 示例生成
- [ ] 更稳定的引用格式

### V2 各层升级

#### 1. 文档层

- 新增 HTML 文档支持（Sphinx / MkDocs 生成的站点）
- 新增 FAQ、错误码文档支持

#### 2. 结构化层

- 元数据扩展：`version`、`product`、`http_method`、`endpoint`、`error_code`
- 版本信息提取与标注

#### 3. 检索层

- 混合检索：向量 + BM25 关键词检索
- Qdrant 元数据过滤（按 version / doc_type）
- Cross-Encoder Reranker 重排序
- 召回 top-10~20 → Rerank → 取 top-3~5

#### 4. 回答层

- 意图识别路由（LLM 分类）：
  - API 查询 → 结构化回答
  - 概念解释 → 通俗化回答
  - 错误排查 → 分步骤排查
  - 版本对比 → 对比表格
- 引用格式标准化：`[文档名 > 章节名 | 版本号]`

#### 5. 交互层

- 支持请求参数指定：`version`、`product`、`doc_type`
- 流式响应（SSE）

#### 6. 评估层

- 扩展测试集，覆盖跨版本对比问题
- 检查是否混用不同版本内容

### V2 典型问题

- v1 和 v2 这个字段有什么区别
- Python 示例怎么写
- 为什么我会报这个 401
- 哪个接口适合做这个事情

## V3：真正可用的 API 文档助手

> **核心命题**：不是一个只会摘抄文档的 RAG，而是一个真正懂技术文档使用方式的助手。

### V3 新增能力

- [ ] 多轮上下文记忆
- [ ] 版本迁移问答
- [ ] 错误排查树
- [ ] 根据目标推荐接口
- [ ] 自动生成 curl / Python / JS 示例
- [ ] 同一个问题下给出"最小可运行示例"
- [ ] 识别用户是在问"概念"还是在"排障"
- [ ] 文档增量更新（监听变更 / 增量 ingest）
- [ ] 离线评估回答质量（自动化指标）

### V3 各层升级

#### 3. 检索层

- 多路召回策略优化
- Query 改写 / 扩展

#### 4. 回答层

- 最小可运行示例生成
- 多语言代码示例（curl / Python / JS）

#### 5. 交互层

- 多轮对话记忆（LangGraph 状态管理）
- 追问自动补全上下文
- 对话历史管理

#### 6. 评估层

- 自动化评估管道
- 检索准确率 / 回答忠实度 / 引用正确率指标
- 回归测试：新功能不影响已有场景
